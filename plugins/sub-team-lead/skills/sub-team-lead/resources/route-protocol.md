# 라우팅 프로토콜

> Phase 2: 분류된 전문가에게 요청을 라우팅하는 전략을 결정한다.

---

## 개요

Phase 1에서 식별된 전문가(들)에 대해 최적의 라우팅 전략을 결정한다.

## 라우팅 전략 결정

### 단일 전문가 (pattern = "single")

신뢰도 ≥ 0.9이면 직접 위임:
- Sister-skill invoke 메시지 구성
- 사용자에게 "→ {expert}에게 위임합니다" 알림
- Phase 3 Coordinate로 이동

### 멀티 전문가 (pattern = "multi")

전문가 간 의존성 분석:
1. **의존성 그래프 구성**: 전문가 A의 출력이 전문가 B의 입력인가?
2. **순차 파이프라인**: 의존성 있음 → A → B → C 순서 결정
3. **병렬 팬아웃**: 의존성 없음 → 동시 실행
4. **혼합**: 일부 순차 + 일부 병렬

### 패스스루 (기존 스킬 정확 매치)

사용자가 특정 스킬을 명시적으로 지정한 경우:
- `sub-api-designer: POST /orders` → sub-api-designer 직접 호출
- team-lead 오버헤드 없이 바로 전달

### 에스컬레이션 기준

| 조건 | 행동 |
|------|------|
| 분류 불가 (confidence = 0) | 사용자에게 명확화 요청 |
| 전문가 실행 실패 | 대체 전문가 시도 또는 사용자 알림 |
| 전문가 결과 불충분 | 추가 전문가 보강 투입 |

## 라우팅 결정 기록

모든 라우팅 결정은 dispatch-report에 기록:
- 어떤 전문가에게 왜 위임했는지
- 어떤 전략(순차/병렬)을 선택했는지
- 예상 실행 순서
